# opsctl Security Guide

`opsctl` is a local safety gate, not a privileged control panel. Keep AI tools on the unprivileged side of the system and make `opsctl` the source of deployment facts.

## Trust Boundaries

- AI tools may read registry facts, inspect deploy gates, inspect backup dry-run readiness, analyze projects, create draft plans, run preflight, request approval, and inspect snapshots.
- AI tools must not receive arbitrary shell, Docker remove, Docker volume delete, Docker system prune, or Caddy overwrite tools through MCP.
- Human approval remains outside MCP. Use `opsctl approve` or `opsctl reject` from an operator shell.
- Mutating CLI paths acquire a global lock under the opsctl state directory. This serializes deploy execution, deploy resume execution, helper execution, snapshot creation, rollback stage/restore, backup execution/prune, approval requests, and approval decisions across concurrent AI tools.
- `opsctl deploy --execute` is CLI-only and only runs opsctl-generated typed operations after a ready dry-run and the printed approval token.
- `opsctl project deliver --execute` is a CLI-only orchestration path for trusted Git hooks/CI. It requires an independently approved, unexpired `automatic_delivery` authorization whose exact origin, branch, canonical project path, service, environment, Profile, statefulness class, and required scopes match the current immutable trigger. It creates and verifies a complete snapshot before invoking the same typed deploy executor; MCP exposes neither authorization creation nor delivery execution.
- `opsctl deploy-resume --dry-run` reads a failed deploy journal and reports resumable operations. `opsctl deploy-resume --execute` is CLI-only and requires an approved journal-specific `deploy_resume.<journal_id>` approval record plus the printed resume token.
- `opsctl rollback --restore` is an operator-only CLI path gated by the approval token printed by `opsctl rollback --dry-run`. MCP does not expose rollback restore.
- `opsctl install-check`, MCP `install_check`, and `opsctl://install/check` are read-only layout checks.

## File Permissions

Recommended Debian layout:

```text
/usr/local/bin/opsctl        0755 root:root
/srv/server-registry         0750 root:opsctl
/srv/server-registry/*.yml   0640 root:opsctl
/srv/server-registry/plans   0750 root:opsctl
/srv/server-registry/approvals 0750 root:opsctl
/var/lib/opsctl              0700 opsctl:opsctl
/var/lib/opsctl/opsctl.db    0600 opsctl:opsctl
/var/lib/opsctl/audit.log    0600 opsctl:opsctl
```

If multiple AI tools run under different Unix users, put them in a narrow group that can read the registry but cannot write approvals unless that is intentional.

## Secret Handling

MCP responses pass through recursive redaction for common sensitive fields such as password, token, secret, API key, cookie, authorization, and private key values.

Project analysis lists `.env` keys only. It must not print `.env` values.

Do not store raw secrets in the registry. Store paths, service names, and environment variable names instead.

Phase 115 signing may read a private key from a direct child of systemd's `CREDENTIALS_DIRECTORY`. The credential name is a restricted identifier, the directory must be absolute and not a symlink, and group/other-readable credential files are rejected. Private key bytes are zeroized after use and are never returned through CLI, TUI, or MCP.

Evidence public keys can be registered with expiry and later revoked. Once the trust store exists, unregistered, expired, revoked, or fingerprint-mismatched keys cannot create trusted signatures. Signed audit checkpoints bind the current audit-chain head; `evidence-verify-all` verifies the chain, manifests, and checkpoints together. The packaged evidence-verification service runs this exact read-only command as the `opsctl` state owner because immutable audit entries can reference root-owned release archives and operator-home artifacts. Only the managed state directory is writable, and the process receives only `CAP_DAC_READ_SEARCH` so it can read those immutable inputs without root or a write-bypass capability. It forbids privilege escalation and has no execution or approval surface; this avoids weakening private directory permissions merely to read historical evidence.

Evidence archive export accepts only a valid trusted signed bundle and an active registered Restic/rustic repository, and it verifies the audit chain before external mutation. opsctl does not configure or attest bucket Object Lock, retention mode, legal hold, or other WORM controls; those policies must be independently enforced and audited.

Phase 119 retention imports accept only bounded, strictly parsed, signed attestations for an active repository. A valid attestation must be fresh, retain data into the future, state that Object Lock and independent verification are enabled, and contain two distinct valid reviewer identifiers. The resulting claim is deliberately limited to `operator_attested_external_policy`; opsctl does not independently prove provider enforcement. Archive drills can restore only snapshots previously recorded by opsctl's evidence archive journal, use generated non-production directories, verify the relocated bundle signature, and remove the generated tree.

Application recovery uses generated internal-only Docker networks and never publishes host ports. Profiles contain only allowlisted environment variable names, fixed engine host/port overrides, safe localhost HTTP paths, and resource limits. Images must be version-pinned and already present locally; arbitrary Compose, commands, production domains, and image pulls are not accepted.

## Snapshot Integrity

Real snapshot creation writes SHA-256 checksums for generated artifacts into `manifest.yml`.

The snapshot scope is intentionally limited:

- registry archive
- best-effort server state JSON
- project analysis JSON
- registered database dump artifacts when the dump target is safe
- registered Docker volume archives when the mountpoint is safe
- Caddyfile copy when visible
- rollback plan

Database dumps and Docker volume archives are best-effort file artifacts, not full VM snapshots. A partial snapshot must not be treated as a full rollback guarantee.

`opsctl snapshot-inspect`, MCP `inspect_snapshot`, and MCP `opsctl://snapshot/{snapshot_id}` are read-only local snapshot manifest reads. They do not restore files, extract archives, validate archive contents, or prove that rollback will succeed.

`opsctl snapshot-verify` and MCP `verify_snapshot` are also read-only. They validate the local snapshot id, refuse artifact paths outside the snapshot root, refuse symlink/non-file artifact reads, and compare each declared artifact against the SHA-256 recorded in `manifest.yml`. They do not extract archives, inspect archive member contents, restore files, execute rollback, or prove that databases and Docker volumes are application-consistent.

`opsctl snapshot-archive-inspect` and MCP `inspect_snapshot_archive` are read-only registry archive checks. They require the `registry_archive` artifact checksum to be verified first, then stream the `registry.tar.zst` tar members without extracting them. They reject unsafe member paths, unsupported entry types, oversized files, oversized total unpacked size, and unreadable/corrupt archive data. They still do not restore files, write to disk, execute rollback, or prove application-level consistency.

`opsctl snapshot-volume-archive-inspect` applies the same read-only archive checks to captured Docker volume archives. It streams archive members without extracting them and does not restore volume data.

`opsctl snapshot-coverage` is a read-only coverage report over registry snapshot records and local snapshot manifests. It does not create, restore, delete, or verify remote backup artifacts.

## Audit Integrity

Every CLI command and MCP tool call records an audit event in SQLite and JSONL where possible.

`read_server_context` reports a lightweight JSONL integrity scan. Invalid JSONL lines are surfaced as warnings because they may indicate manual edits, truncation, or tampering.

Use `opsctl audit --limit 20 --json` for a recent event query with the same integrity report. Invalid JSONL rows are not returned as events, but their line numbers remain visible in `integrity.invalid_lines`.

MCP resource `opsctl://audit/tail` exposes the same read-only audit tail for AI clients.

`opsctl install-check`, MCP `install_check`, and `opsctl://install/check` validate the local registry/state layout, required registry files, symlink safety, basic permissions, and typed registry loading. They do not create directories, change permissions, write registry files, execute deployment, or repair installation problems.

On Unix, JSONL audit writes refuse symlinked audit log paths.

MCP resource templates are strict allowlist lookups for registry service ids, port numbers, domain hosts, snapshot ids, and backup plan service ids. They do not expose `file://` or arbitrary filesystem paths.

Schema resources are embedded in the `opsctl` binary. `opsctl registry export-schema <name>` and MCP `opsctl://schema/{name}` do not read arbitrary schema paths from disk.

`opsctl registry validate` validates registry YAML against those embedded schemas with JSON Schema HTTP and filesystem reference resolver features disabled. Schema errors block typed registry loading and doctor checks.

`opsctl registry normalize` is a dry-run by default. With `--execute`, it loads the existing registry, normalizes schema-compatible YAML formatting, omits null optional fields, maps legacy observed/adopted port source values back to `observed`, and normalizes CLI exposure aliases such as `local` to registry values such as `localhost`. It refuses symlinked registry files, writes through restrictive temporary files, acquires the global mutating-command lock, and does not add, delete, stop, deploy, back up, or clean up any service resource.

`opsctl backup readiness` and `opsctl backup plan <service-id> --dry-run` only create dry-run readiness reports and Restic/rustic command previews. They report required environment variable names and check whether they are set, but they do not print or persist values, execute backup commands, create database dumps, initialize repositories, prune repositories, or restore data.

`opsctl backup repo-init <repository-id>` is a dry-run review by default. `--execute` requires the printed `repo-init:<repository-id>` approval token and runs only `restic init` or `rustic init` for an active registry repository with all required environment variables present. It writes repository metadata to the configured backend, but it does not write backup history, repository check history, restore drill history, or application data.

`opsctl backup run <service-id> --execute` is the controlled CLI execution path. It only runs planner-generated database dump and Restic/rustic commands for a ready plan, avoids persisting stdout/stderr contents, and appends local backup history. External database dump scripts are allowed only when `backups.yml` declares `adapter`, `script`, and `working_dir`, and the same `adapter/script` pair is declared in `services.yml deployment.build`; opsctl executes only `npm|pnpm|bun run <script>` without a shell and requires the script to write `OPSCTL_BACKUP_DUMP_OUTPUT`. `opsctl backup check` is controlled repository execution. `opsctl backup prune` requires an explicit approval token.

`opsctl backup restore-plan <service-id> --repository-snapshot <id> --restore-dir <dir>` is dry-run only and prints the restore approval token. `opsctl backup restore ... --execute --approval-token <token>` re-plans the restore, requires the token, and only runs `restic restore` or `rustic restore` into an absolute staging directory that is empty or safely creatable. `opsctl backup drill <service-id> --restore-dir <dir>` resolves the latest successful registered snapshot for the target, then uses the same restore planner/executor. Its `--scheduled` execution mode is restricted to `/var/lib/opsctl/restore-drills/...` for systemd timer use and creates a unique `run-*` child staging directory for each execution. `opsctl backup drill-suite` applies that same drill planner to several services; execution uses scheduled-style restore directories and remains CLI-only. These commands refuse symlinked paths, parent traversal, protected roots, and registered service roots/data/include/dump paths. They do not restore directly over production paths and are not exposed through MCP.

`opsctl backup drill-cleanup` is a dry-run review by default. `--execute` only deletes immediate `/var/lib/opsctl/restore-drills/<service>/run-*` directories selected by `--keep-days` and `--keep-count`; it skips symlinks, non-directories, and non-`run-*` names. It is intended for scheduled restore drill staging cleanup, not arbitrary file deletion.

`opsctl backup timer plan`, `opsctl backup timer install`, `opsctl backup timer status`, `opsctl backup timer monitor`, `opsctl backup timer alert`, `opsctl backup timer alert-test`, and `opsctl backup timer alert-configure` standardize the packaged backup timer workflow. `install` is still a dry-run unless `--execute` is supplied; execution only runs `systemctl enable --now <planned-timer-unit>` for units derived from the registry's active backup targets and repositories. `monitor` is read-only: it reads timer status, service result fields, optional recent warning/error journal lines, and registered backup/check/drill history. It reports alert candidates and configured alert sinks from `policies.yml`. `alert-configure` is a dry-run unless `--execute` is supplied; execution only adds or updates one `timer_alerts` entry in `policies.yml`, stores the target environment variable name, and never stores or prints the URL/token/recipient value. Active sink execution is blocked unless the named target environment variable is present; disabled sinks can still be registered without a target value. `alert` is dry-run unless `--execute` is supplied and only sends when current failures produce alert candidates. `alert-test` is also dry-run unless `--execute` is supplied; it sends a synthetic test payload to active alert sinks, optionally limited by `--sink-id`, so operators can verify delivery when no real failures exist. Executed alerts support active `webhook`, `ntfy`, `telegram`, and `email` sinks from `policies.yml`; target values are read from named environment variables and are not printed in JSON/text reports. HTTP alert sinks use curl config/payload temp files with restrictive permissions so token-bearing URLs are not passed as process arguments; email delivery uses `sendmail -t` without a shell. MCP does not expose alert configuration or alert execution.

The global mutation lock is fail-fast by default. Packaged backup/check/drill and selected recovery units set `OPSCTL_LOCK_WAIT_SECONDS` to opt into bounded queuing; the accepted value is an integer from zero through 21,600 seconds. A waiting process neither deletes nor rewrites the lock, reads at most 4 KiB of holder metadata for a timeout diagnostic, and is still bounded by its systemd unit timeout. Repository checks acquire the same lock because they append check history. Setting the wait variable outside reviewed scheduled units is an explicit operator choice, not an automatic CLI behavior.

`opsctl backup onboarding-check` is a read-only production readiness summary. It lists the backup run, repository check, restore drill, observed import check, and promotion dry-run steps needed for a real production registry promotion. It does not run backup jobs, contact repositories, create restore drills, or promote imports.

`opsctl backup s3-smoke --execute` is a controlled S3-compatible storage smoke test. It uses rclone environment configuration, requires access key and secret key values through named environment variables, writes one generated payload object under a unique or supplied prefix, downloads it for hash/content verification, lists the prefix, deletes the object, and removes empty prefix markers when possible. Its report prints environment variable names, endpoint, provider, bucket, prefix, operation status, and sanitized diagnostics only; it does not print credential values, write registry history, initialize Restic repositories, run backup jobs, or prove application-level restore readiness. A successful smoke test is useful for credential/endpoint onboarding, but production deploy gates still require `backup repo-init` when the repository is new, followed by real `backup run`, `backup check`, and `backup drill` records.

`/usr/share/opsctl/scripts/production-onboarding-check.sh` defaults to that same read-only check. It can load an operator-managed credential file through `OPSCTL_BACKUP_ENV_FILE` or `/etc/opsctl/restic.env`. Setting `OPSCTL_ONBOARDING_APPLY=1` makes the script execute only explicit `OPSCTL_ONBOARDING_SERVICES` and `OPSCTL_ONBOARDING_REPOSITORIES` through typed `opsctl backup run`, `opsctl backup check`, and `opsctl backup drill-suite --execute` commands. `OPSCTL_ONBOARDING_REPO_INIT=1` additionally runs approval-token-gated `opsctl backup repo-init` first and should only be used for new repositories. It then runs `registry import-check --scan-observed` and `registry promote-import --dry-run --scan-observed`. It still does not automatically promote the import.

`opsctl backup history`, MCP `backup_history`, and `opsctl://backup/history` read registered backup result records, repository check records, and restore drill records from `backups.yml`. They do not contact remote repositories while reading history. A production `before_deploy` target is ready only when the latest backup record, latest repository check record, and latest restore drill record are all successful, recent, not future-dated, and free of limitations.

`opsctl backup check <repository-id>` is the controlled path that executes `restic check` or `rustic check` and appends a `repository_checks` record. `opsctl backup restore ... --execute` appends a `restore_drills` record after a successful staging restore and verification. These writes are local registry facts; they still rely on the operator to run checks against the intended repository and to protect backup credentials.

`policies.yml` can declare temporary `public_data_port_exceptions` for intentionally exposed database/cache ports. Each active exception must name the port id, owner, reason, expiry, and mitigation. `opsctl doctor` suppresses the base `public_data_port` warning only while the exception is active and unexpired; expired, invalid, or dangling exceptions produce their own warnings.

`opsctl status` includes compact deploy gate, backup readiness, backup history, and snapshot coverage summaries. These fields reuse the same dry-run readiness, registered-history, and registered-snapshot boundaries; they do not execute backup commands, create snapshots, restore data, or inspect remote repositories.

`opsctl deploy-gates`, MCP `deploy_gates`, and `opsctl://deploy/gates` combine backup readiness, registered backup history, registered snapshot coverage, and registered timer-health failure thresholds into one before-deploy gate report. They are read-only/dry-run fact sources. They do not execute deployment, backup, snapshot, restore, prune, check, or database dump commands, and they summarize counts instead of printing secret values or artifact paths.

`opsctl rollback --stage-dir <new-dir>` extracts the verified registry archive into a new staging directory and does not touch production paths. `opsctl rollback --restore --approval-token <token>` can replace the registry after dry-run conflict checks and creates a rollback backup archive first. Caddy config and Docker volume restore are opt-in through `--restore-config` and `--restore-data`.

`opsctl request-deploy-execution` creates a requested `deploy_execution` approval record for a ready deploy dry-run. `opsctl deploy --execute` requires an approved `deploy_execution` approval record and the approval token printed by `opsctl deploy --dry-run`. It re-runs preflight and snapshot verification before execution, writes a deploy journal under the state directory, stops at the first failed operation, and only executes typed operations generated by `opsctl`. Supported execution operations are managed Caddy route write, generated typed Caddy route snippet file write, Caddy validate/reload, Docker Compose up, safe static-site sync, package-manager build steps (`npm`, `pnpm`, `bun`), Laravel artisan cache/optimize steps, systemd service reload/restart, exact allowlisted migration commands, optional post-deploy health checks, and registry write-back. Generic raw file writes are intentionally reported as unsupported until dedicated typed adapters exist.

Post-deploy health checks run only when a plan sets `changes.health.enabled: true`. They verify declared Docker containers with `docker inspect`, reserved localhost ports with TCP connect, declared Caddy hosts with a local HTTP Host probe, and static-site target files. Failed health checks are written into the deploy journal and stop before registry write-back. The journal includes rollback suggestions, but rollback execution still requires the normal rollback dry-run and approval flow.

Typed Caddy snippet writes accept only `kind: caddy_route_snippet` with `host` and `upstream` params. The adapter generates file content itself, requires an absolute safe target path with a `.caddy` extension, refuses symlink parents or symlink targets, and refuses to overwrite an existing file unless it already contains the opsctl typed-file marker.

Migration execution does not invoke a shell. It only accepts exact commands currently allowlisted by the migration adapter: `npm run migrate`, `npm run db:migrate`, `pnpm run migrate`, `pnpm run db:migrate`, `pnpm exec prisma migrate deploy`, `bun run migrate`, `bun run db:migrate`, `php artisan migrate --force`, `php artisan migrate --force --no-interaction`, and `composer run migrate`.

Build, Laravel, and systemd deploy adapters do not invoke a shell. Build steps accept only `npm`, `pnpm`, or `bun` plus a safe script name. Laravel steps are generated from booleans for `optimize`, `config:cache`, `route:cache`, and `view:cache`. Systemd steps accept only `.service` unit names with safe characters and `reload` or `restart`.

`opsctl caddy-routes` is a read-only Caddyfile inspection command. It reports opsctl-managed route markers, simple unmanaged site blocks, malformed managed blocks, and a shallow Caddyfile import graph for exact, snippet, and dynamic/glob `import` directives. With `--adapt`, it calls `caddy adapt` read-only, normalizes JSON HTTP routes, normalizes wildcard hosts and path-prefix matchers, summarizes matcher kinds, recursively summarizes handler chains, computes a conservative specificity score, reports duplicate/overlapping route matchers, highlights risky route-priority overlap, and reports TLS automation subject conflicts. With `--admin`, it reads `GET /config/` only from a loopback Caddy Admin API endpoint (`127.0.0.1`, `localhost`, or `[::1]`) and returns summary counts for apps, HTTP servers, routes, and TLS policies. The JSON report includes a `management` summary for managed/unmanaged/import/conflict counts and safe next actions, but it explicitly reports `admin_api_write_supported: false`. It does not write config, expand glob imports, read imported file contents, call non-loopback Admin API endpoints, reload services, return full Admin API config, or mutate routes.

`opsctl registry drift list`, `opsctl registry drift groups`, `opsctl registry drift suggest`, `opsctl registry drift governance`, `opsctl registry drift review export`, `opsctl registry drift cleanup-plan`, `opsctl registry drift cleanup-request export`, `opsctl registry drift cleanup-request verify`, `opsctl registry drift cleanup-request triage`, `opsctl registry drift cleanup-request dashboard`, `opsctl registry drift cleanup-request worklist`, `opsctl registry drift cleanup-request evidence`, `opsctl registry drift cleanup-request evidence-plan`, `opsctl registry drift cleanup-request execution-plan`, `opsctl registry drift cleanup-request execution-gate`, `opsctl registry drift cleanup-request approval-pack`, `opsctl registry drift cleanup-request volume-ownership`, MCP `registry_drift_cleanup_evidence_plan`, MCP `registry_drift_volume_evidence_plan`, and `opsctl registry drift explain` are read-only views over `--scan-observed` style drift findings unless an explicit CLI `--execute` is supplied for evidence collection. They apply active `policies.yml` drift ignore rules, report ignored findings separately, summarize findings by code/kind, group findings by likely resource ownership prefix, export grouped review YAML, and surface expired or invalid ignore rules as limitations. `review export` includes ownership evidence, service candidates, cleanup risk, exact-match requirements, and resource fingerprints to make human review easier, but those fields do not authorize cleanup or adoption by themselves. `ownership` adds process/container/volume evidence, resource fingerprints, confidence counts, and a suggested review order, but still does not prove business ownership. `governance` summarizes active drift, ignored drift, groups, cleanup candidates, public-listener cleanup candidates, data-risk cleanup candidates, and next operator actions. `cleanup-plan` marks cleanup candidates and risk only; it never generates destructive Docker/systemd/kill commands. `cleanup-request export` converts that plan into a human-review YAML document, `cleanup-request verify` validates reviewed status, owner, reason, public-listener risk, data-risk flags, and confirms no destructive commands were embedded, and `cleanup-request triage`, `cleanup-request dashboard`, and `cleanup-request worklist` summarize unknown ownership decisions plus needs_cleanup approval/evidence gaps without writing files. `cleanup-request approval-pack` is also read-only; it packages each selected item with collected evidence, missing approval fields, Docker volume backup/restore evidence requirements, current/stale status, safety checklist entries, and a `cleanup-request mark ... --approval-status approved ... --execute` template for a human operator to review. That template is not executed by the approval-pack command and does not clean resources. `cleanup-request evidence-plan` and MCP `registry_drift_volume_evidence_plan` expose grouped Docker volume evidence planning, batch steps, backup snapshot gaps, and restore drill gaps without collecting evidence, approving cleanup, or executing removal. `cleanup-request volume-ownership` is read-only as well; it classifies Docker volume cleanup items by anonymous hash name, named volume, attached containers, service candidates, and missing backup/restore proof. It may print dry-run review/adopt/mark command templates, but it does not execute them, does not approve items, and does not read or remove volume contents. `cleanup-request evidence --execute` writes only `collected_evidence` and `evidence_collected_at` into selected review items after collecting current ownership evidence; it does not set `approved`, change owners, or clean resources. `cleanup-request execution-plan` validates the same reviewed file against pre-execution evidence gates such as cleanup strategy, exact resource id, approval expiry, maintenance window, rollback plan, backup snapshot evidence, and restore drill evidence. `cleanup-request execution-gate` exposes `auto_cleanup_supported: false`, `destructive_executor_status: not_implemented_by_design`, and `destructive_execution_supported: false` so clients cannot mistake the handoff path for a deleter. Cleanup runbooks set `safe_to_automate: false` and `requires_separate_destructive_approval: true` for each step. `cleanup-request request-execution` creates an audited approval request only. `cleanup-request execute --execute` does not delete resources; it requires an approved cleanup-execution approval, the printed token, a human reason, the global opsctl lock, and a fresh current-drift revalidation that every ready item still matches its exact observed resource id before writing a JSONL manual handoff record. `opsctl registry drift service-add` creates one empty service skeleton in `services.yml` only when `--execute` is supplied; it requires a human reason, writes no ports/containers/volumes/domains/systemd units, and is intended only to create an explicit target for later single-resource adoption. `opsctl registry drift review apply` validates a human-edited review YAML file and stays dry-run unless `--execute` is supplied; execution can batch approved `ignore` and `adopt` actions through the same single-item policy paths and global lock. `opsctl registry drift ignore` adds one expiring ignore rule to `policies.yml` only when `--execute` is supplied; it requires a human reason and expiry, writes `drift-ignores.jsonl`, and never deletes or stops the observed resource. `opsctl registry drift adopt` can adopt one explicit observed target at a time for `port`, `caddy-domain`, `docker-container`, `compose-project`, `docker-volume`, or `systemd-unit`. It requires an existing service id, stays dry-run unless `--execute` is supplied, refuses ambiguous `auto` matches, requires `--reason` for execution, records optional operator notes and review status, writes a separate `drift-adoptions.jsonl` journal under the state directory, writes only the relevant registry YAML fields, and annotates Caddy domains / Docker volumes / systemd contracts as observed facts needing manual review. Multi-file registry adoption snapshots touched files first and rolls back earlier file writes when a later adopt write fails. It does not infer deployment commands, delete resources, execute Docker/Caddy/systemd, prove ownership, or automatically take over all drift findings.

`opsctl deploy-journals`, `opsctl deploy-journal-inspect`, MCP `list_deploy_journals`, MCP `inspect_deploy_journal`, `opsctl://deploy/journals`, and `opsctl://deploy/journal/{journal_id}` are read-only local journal reads. They validate journal ids and refuse symlinked journal directories plus oversized, non-file, or symlinked journal files. They do not execute deployment, modify registry files, or restore snapshots.

`opsctl deploy-resume ./deploy-plan.yml --journal <journal-id> --dry-run` is read-only. It only accepts failed journals, verifies the current plan id and operation sequence still match the journal, refuses journals that already wrote registry state, and reports the failed operation plus remaining operations. It does not run Docker, Caddy, migrations, registry writes, rollback, or helper operations.

`opsctl request-deploy-resume` creates a requested `deploy_resume.<journal_id>` approval record for a resumable failed journal. `opsctl deploy-resume --execute --approval-token <token>` requires that journal-specific approval scope, re-runs the current deploy readiness checks with the original snapshot id, verifies the printed resume token, and writes a new deploy journal for the resumed operation list. It does not mutate the original failed journal. MCP exposes only `deploy_resume_dry_run`, not resume execution.

`opsctl helper run-deploy-operation` executes one typed deploy operation by order and is the intended sudoers allowlist boundary for future privileged setups. Do not grant AI users direct sudo access to Docker, Caddy, systemctl, shell, or file deletion commands.

`opsctl helper sudoers-check --path /etc/sudoers.d/opsctl-helper` is a read-only policy validator. It checks the file type, permissions, helper allowlist content, forbidden broad command patterns, and `visudo -cf` syntax when `visudo` is available. The repository's `scripts/install-sudoers.sh` remains dry-run unless `OPSCTL_SUDOERS_APPLY=1` is set.

Backup target `max_age_hours`, `repository_check_max_age_hours`, and `restore_drill_max_age_hours` are local freshness policies over registered backup, repository check, and restore drill records. Stale, future-dated, invalid, failed, or limitation-bearing records block production deploy gates for linked `before_deploy` services.

`opsctl backup restore --execute` and `opsctl backup drill --execute` verify staged restore output after the repository restore command succeeds. By default they check restored file counts, hash samples, compressed dump presence, and plain SQL dump plausibility. Setting `OPSCTL_RESTORE_DB_IMPORT_CHECK=1` additionally runs plain or zstd-compressed MariaDB/MySQL/Postgres dumps through isolated no-network Docker containers. This executes restored SQL only inside temporary containers and does not connect to production databases.

Production preflight uses the same dry-run planner, registered backup history report, and registry-backed snapshot coverage report for services with `backup_policy: before_deploy`. This can block deployment readiness when the backup plan is not ready, when backup history is missing, failed, stale, future-dated, or invalid, or when registered snapshot coverage is missing, incomplete, missing required scope, invalid, or limited. `opsctl deploy-gates` may include structured blocked details and remediation command hints for stale or failed backup targets, but those hints are report text only and are not executed by the gate check. Preflight itself still does not execute backup commands, create snapshots, inspect archives, restore data, or inspect remote repositories.

MCP `deploy_gates`, `caddy_routes`, `registry_drift_list`, `registry_drift_explain`, `install_check`, `list_deploy_journals`, `inspect_deploy_journal`, `deploy_resume_dry_run`, `backup_doctor`, `backup_readiness`, `backup_history`, `backup_plan`, `backup_onboarding_check`, `backup_timer_plan`, `snapshot_coverage`, `inspect_snapshot`, `verify_snapshot`, `inspect_snapshot_archive`, `request_deploy_execution`, `opsctl://deploy/gates`, `opsctl://caddy/routes`, `opsctl://install/check`, `opsctl://deploy/journals`, `opsctl://deploy/journal/{journal_id}`, `opsctl://backup/doctor`, `opsctl://backup/readiness`, `opsctl://backup/history`, `opsctl://backup/plan/{service_id}`, `opsctl://snapshot/coverage`, and `opsctl://snapshot/{snapshot_id}` expose backup and rollback facts to AI clients. Deploy gates, backup readiness, backup plan, backup onboarding, backup timer planning, registry drift views, and deploy resume dry-run tools/resources are always dry-run/read-only and follow the same no-execution boundary as the CLI command. Caddy routes, install checks, deploy journal reads, snapshot coverage, inspect, verify, and archive inspect tools/resources are read-only and do not create or restore snapshots. `request_deploy_execution` creates an approval request only and redacts execution-token-like output. MCP does not expose `deploy --execute`, `deploy-resume --execute`, `helper run-deploy-operation`, `backup run`, `backup restore`, `backup check`, `backup prune`, `backup timer install --execute`, `backup timer alert --execute`, `backup timer alert-test --execute`, `registry normalize --execute`, `registry drift service-add --execute`, `registry drift adopt --execute`, cleanup request execution handoff, `snapshot-volume-archive-inspect`, or `rollback --restore`.

The TUI dashboard includes deploy gate, backup readiness, backup restore readiness, backup history, snapshot coverage, observed drift, drift ownership review counts, deploy adapter, registry promotion backup, deploy journal, and install-check summaries. The TUI can approve or reject selected pending approval records, but it does not execute deploy, backup, restore, prune, drift cleanup, deploy resume, or rollback restore operations. MCP server context also includes the deploy gate summary.

## Volume Evidence Resolution and Protection

`cleanup-request evidence-resolve` never changes approval status and never invokes Docker deletion. Its `--execute` path only copies fresh, exact backup and restore identifiers into the review YAML. Prefix-only service guesses, stale cleanup items, multiple exact owners, changed current content, stale records, backup targets that do not cover the volume, failed or limited restores, and restore drills without checked files cannot be written automatically. `--verify-repository` runs only a read-only snapshot listing and requires the exact snapshot and ownership tags; MCP never enables this remote access.

`backup volume-protect run --execute` is a non-destructive CLI-only protection path for current orphan volumes. It rejects mounted volumes, volumes with service candidates, symlinked or non-directory mountpoints, inactive or unsupported repositories, missing repository environment, existing restore directories, `/` as a restore root, symlinked restore ancestors, and restore roots overlapping the source or registered service/data paths. It does not stop containers, approve cleanup, prune repositories, or remove volumes. A successful run requires the restored tree fingerprint and detected database features to match before writing `volume-protect.jsonl` and cleanup evidence. The bounded fingerprint includes relative path, type, size, mtime, and sampled content; it is a change detector, not a full forensic hash of every byte.

Lifecycle events are appended to a size-limited, no-follow JSONL journal. Resume requires a recorded successful snapshot and creates a new isolated restore directory. Staging cleanup only considers non-symlink directories recorded by that journal beneath an absolute, non-root restore root; it defaults to dry-run and honors both age and newest-count retention. Batch execution is serial, defaults to dry-run, rejects mounted/owned/unreadable/truncated candidates, and enforces explicit item and byte limits. Failure alerts are opt-in and use only active registry-configured sinks.

Protection campaigns preserve serial execution and add free-space reserve, duration, item, byte, and failure fuses. Successful targets are skipped on resume; a failed target reuses its recorded snapshot only with the original minimum verification strength. `feature` proves recognizable engine structure and `integrity` adds engine metadata/integrity checks. `boot` requires either the read-only SQLite integrity/open check or a valid exact-volume recovery profile whose preloaded, version-pinned PostgreSQL, MySQL/MariaDB, Redis, or MinIO image reaches readiness from a temporary working copy. Recovery containers use `--network none`, publish no ports, use `--pull never`, have bounded memory/CPU/PIDs/time, and never mount the original restore writable. Missing Docker/images or failed boot/probes block evidence.

`cleanup-request handoff-pack --execute` requires an already recorded approved manual handoff and a ready execution plan. It writes a read-only (`0400` on Unix), expiring SHA-256 sealed manifest bound to the complete cleanup request and the exact request hash recorded by the handoff. `manifest-status` rejects tampering, expiry, request changes, and missing/mismatched handoff records. `reconcile --execute` only appends finalize evidence for sealed items that the existing current-drift comparison reports absent; it cannot stop or delete resources. Journal maintenance archives removed lines with a SHA-256 sidecar before atomically rewriting known volume-protect journals. Operational failure alerts use a local cooldown state, respect sink severity thresholds, and emit recovery only after a previously failed run succeeds.

Evidence key generation is create-new and stores private keys without group/other access under the state directory. Detached Ed25519 signatures bind the complete immutable manifest bytes to a trusted rotation key; `--require-signature` makes manifest status and reconciliation fail closed until strict verification succeeds. Run, campaign, handoff-manifest, signature, and reconciliation artifacts feed a hash-linked audit journal. Mutable journals must match their previously audited hash before another append. Audit bundles are create-new/read-only exports intended for transfer to operator-managed WORM storage; opsctl does not claim that ordinary local filesystem permissions are WORM.

MCP exposes only evidence resolution previews, cleanup workflow/finalize/handoff reads, volume-protect history, lifecycle/campaign status, metrics, and manifest validation. TUI workflow filters, campaigns, and metrics are read-only. Neither surface exposes volume protection execution, resume, staging cleanup, batch/campaign execution, journal maintenance, sealing, reconciliation writes, or destructive cleanup.

Phase 117–121 MCP additions (`recovery_qualification`, `evidence_backfill_plan`, `evidence_retention_status`, `evidence_archive_drill_status`, `evidence_key_dr_status`, and `recovery_slo`) are read-only and audited. Retention import, archive drill execution, backfill trend writes, and governance timer installation remain explicit CLI-only paths protected by the global mutation lock. Governance packages install timer units but never enable them automatically.

## Protected Operations

The policy engine blocks or requires approval for:

- port/domain/container/volume conflicts
- protected path writes and deletes
- registered data path writes and deletes
- Docker internal path writes
- production migrations
- missing ready backup plans, ready backup history, or ready snapshot coverage for registered production services that require `before_deploy` backups
- destructive command patterns
- unknown change sections

When uncertain, prefer `blocked` or `needs_approval`.

Automatic delivery does not make stateful recovery implicit. Stateless classification rejects database/migration evidence and stateful Compose. Database classification accepts only PostgreSQL, MySQL/MariaDB, and SQLite managed Node contracts with typed migrations and current registered backup, repository check, restore drill, and snapshot readiness. Missing, stale, unsupported, or ambiguous evidence remains manual-only.
