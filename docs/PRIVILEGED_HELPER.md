# Privileged Helper Design

Phase 32 enables a first controlled execution boundary for deployment operations.

The goal is to prevent `opsctl` from becoming a root shell for AI tools. AI tools should submit deploy plans; `opsctl` converts those plans into typed operations; `opsctl helper run-deploy-operation` may execute one selected operation after preflight, snapshot, approval, and execution-token gates pass.

## Non-Goals

- No arbitrary shell command execution.
- No direct Docker socket exposure to AI tools.
- No direct Caddyfile overwrite from MCP.
- No root shell, interactive shell, or user-supplied command string passthrough.
- No deployment execution before approval and snapshot checks are complete.

## Operation Model

The CLI produces operations such as:

- `PreflightCheck`
- `VerifySnapshot`
- `ReservePort`
- `WriteCaddyRoute`
- `ValidateCaddy`
- `ReloadCaddy`
- `ComposeUp`
- `RunMigration`
- `WriteFile`
- `WriteRegistry`

Each operation has:

- stable `kind`
- target
- explicit argv when a system command is involved
- working directory when relevant
- privilege requirement
- reason

Migration command text is intentionally not echoed in dry-run output. The migration runner executes only exact allowlisted commands without a shell and does not persist stdout/stderr.

`WriteFile` is not a generic file write primitive. The only supported adapter is `template_kind=caddy_route_snippet`, which generates a managed Caddy snippet from `host` and `upstream` params. Raw file content from deploy plans remains unsupported.

## Helper Boundary

The helper accepts a deploy plan, operation order, optional snapshot id, and the approval token printed by `opsctl deploy --dry-run`:

```text
opsctl helper run-deploy-operation ./deploy-plan.yml --operation 4 --snapshot <snapshot-id> --approval-token <token>
```

Validate a reviewed sudoers helper policy with:

```text
opsctl helper sudoers-check --path /etc/sudoers.d/opsctl-helper
```

The helper should reject:

- unknown operation kinds.
- extra argv segments not generated by `opsctl`.
- relative paths for system files.
- paths outside allowlisted roots.
- Docker Compose operations without explicit project names.
- invalid or missing deploy execution approval tokens.

## Allowed System Operations

Initial helper allowlist:

```text
caddy validate --config /etc/caddy/Caddyfile
systemctl reload caddy
docker compose --project-name <name> up -d
npm run migrate
npm run db:migrate
pnpm run migrate
pnpm run db:migrate
pnpm exec prisma migrate deploy
bun run migrate
bun run db:migrate
php artisan migrate --force
php artisan migrate --force --no-interaction
composer run migrate
```

Future operations can be added one by one with tests and audit coverage.

## Sudoers Productionization

The repository includes:

- `templates/sudoers.opsctl.example`
- `scripts/install-sudoers.sh`
- `opsctl helper sudoers-check`

The installer is dry-run by default. It requires `OPSCTL_SUDOERS_APPLY=1` and root before writing `/etc/sudoers.d/opsctl-helper`. It validates the generated file with `visudo -cf` before installation and runs `opsctl helper sudoers-check` after installation when `opsctl` is available.

`opsctl helper sudoers-check` rejects symlinked or missing policy files, group/other writable files, policies that do not mention `opsctl helper run-deploy-operation`, and broad patterns such as `NOPASSWD: ALL`, Docker, shell, `rm`, or `systemctl`.

## Audit Requirements

Every helper execution must record:

- actor
- plan id
- operation id
- operation kind
- target
- argv after redaction
- working directory
- start and finish timestamps
- exit status
- redacted stdout/stderr location

Failed operations must be audited too.

## Current Behavior

`opsctl deploy --dry-run` prints typed operations and an execution approval token when the plan is ready.

`opsctl deploy --execute --approval-token <token>` runs the full typed operation list and writes a deploy journal. `opsctl helper run-deploy-operation` runs one operation by order and is intended for sudoers allowlists. `RunMigration` is supported only for the exact allowlist above. `RunBuild` supports only `npm`, `pnpm`, and `bun` script names generated from the plan. `LaravelOptimize` supports only generated artisan cache/optimize commands. `SystemdService` supports only generated `.service` reload/restart commands. `WriteFile` supports only generated Caddy route snippets; generic raw file writes remain unsupported until dedicated typed adapters exist.

`opsctl deploy-resume --execute --approval-token <token>` is also CLI-only. It requires a failed journal, a current resumable dry-run, an approved journal-specific `deploy_resume.<journal_id>` approval record, and the printed resume token. It writes a new journal for the resumed operation list instead of modifying the failed journal.
