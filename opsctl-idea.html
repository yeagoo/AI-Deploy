为什么 怎么做 部署流程 快照回滚 第一版

轻量、不接管、专门保护服务器

给 AI 部署加一道看得见的安全门。

这不是另一个控制面板，也不是完整的部署平台。它更像服务器门口的值班员： 先看清现状，再检查计划，确认安全后才放行。

看核心思路 看部署过程
DEPLOY GATE 本机使用 · 不开放公网
AI
提交部署想法 例如要启动新项目、绑定域名、使用某个端口。
检查
确认不会撞车 看端口、服务、数据目录、Docker 卷、Caddy 路由。
审批
危险动作先问你 涉及删除、覆盖、迁移、停生产服务时必须确认。
执行
只执行通过的计划 留下记录，方便排查和回滚。
不替代 Docker 只是把风险动作先拦住。
不替代 Caddy 只帮你检查路由是否安全。
不做公网后台 本机 TUI 和本地 MCP 就够。

问题不是 AI 会不会部署，而是它会不会误伤。

一台服务器上已经有多个业务、数据库、缓存、域名和端口。新的项目一部署， 如果没有统一登记和部署前检查，就可能把旧项目挤掉。

1

端口被占

新项目默认用 3000、8080、5432，可能正好是老项目正在使用的端口。

2

数据被碰

数据库卷、上传目录、备份目录如果没有保护，误删一次就是生产事故。

3

路由被覆盖

域名反代一改错，原来能访问的网站就可能突然打不开。

核心思路：不做平台，只做护栏。

它不负责替你决定怎么部署，也不强迫所有项目改成同一种形态。 它只做一件事：让每次部署都先过一遍安全检查。

它要做的事

  • 记录服务器上已有服务和端口。
  • 检查新项目会不会冲突。
  • 把危险动作列出来让你确认。
  • 给 AI 工具提供准确上下文。

它不做的事

  • 不开放公网管理后台。
  • 不接管全部 Docker 和 Caddy。
  • 不让 AI 直接拿最高权限乱跑。
  • 不把服务器改成复杂平台。

一次安全部署，大概分五步。

AI 仍然可以帮你写配置、构建项目、准备部署方案。但真正执行之前， 需要先经过这道门。

STEP 01

读现状

知道哪些端口、域名、容器、数据卷已经在用。

STEP 02

看计划

AI 先提交部署计划，而不是直接执行命令。

STEP 03

查冲突

发现端口、路由、数据目录或容器名冲突就阻断。

STEP 04

问审批

涉及删除、覆盖、迁移、停服务时必须让你确认。

STEP 05

留记录

每次部署都留下历史，出了问题能追溯。

更稳的做法：新项目部署前，先给服务器留一张快照。

快照不是为了让部署变慢，而是为了出问题时能回到一个明确的时间点。 这样第三个应用部署失败，也不会让前两个业务一起不可用。

部署前先保存什么？

不一定每次都克隆整台机器。更实用的是分层保存：配置、部署清单、 数据库、Docker 卷、Caddy 路由和关键目录。

Caddy 配置 Docker Compose 数据库 dump Docker volumes 项目 env 清单 当前端口状态

快照要能回答三个问题

部署前
服务器原来有哪些服务正在跑？入口在哪里？数据在哪里？
部署中
这次计划会改哪些文件、端口、路由、容器和数据卷？
失败后
应该停止什么、恢复什么、重新加载什么，才能回到部署前？
把 AI 当成施工队，不要把总钥匙直接交出去。

施工队可以画图、搬材料、搭架子；但进生产机房、动数据库、改入口路由， 应该先登记、检查、签字。

第一版不用复杂。

先做成一个 Rust 单文件工具：本地打开 TUI 给人看，通过 MCP 给 AI 读， 执行前用规则检查。没有网页后台，也不开放新端口。

✓
服务器登记簿 记录服务、端口、域名、数据目录、备份位置。
✓
部署前检查 先检查 Docker、Caddy、端口、数据卷是否冲突。
✓
危险操作审批 删除、覆盖、迁移、停止生产服务都要确认。
✓
部署前快照 保存配置、路由、数据和回滚说明，失败时能恢复到部署前。
✓
本地 TUI 你 SSH 进去就能看服务器状态和待审批事项。
✓
本地 MCP Codex、Claude Code、opencode 只读上下文，少猜少误操作。
这个页面描述的是轻量方案：它不是 PaaS，也不是控制面板，而是给 AI 部署加上的本机安全闸门。
